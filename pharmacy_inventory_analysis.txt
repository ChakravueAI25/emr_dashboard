PHARMACY INVENTORY SYSTEM ANALYSIS

1. PHARMACY COLLECTION STRUCTURE

Collection definition:
- backend/database.py
- pharmacy_collection = db["pharmacy_medicines"]

Primary medicine model:
- backend/models.py
- class PharmacyMedicine(BaseModel)

Fields used for medicines:
- id: Optional[str]
- name: str
- category: str
- price: float
- stock: int
- description: str
- manufacturer: Optional[str]
- batch_number: Optional[str]
- expiry_date: Optional[str]
- reorder_level: int
- created_at: Optional[datetime]
- updated_at: Optional[datetime]

Fields confirmed in backend GET responses:
- id
- name
- category
- price
- stock
- description
- manufacturer
- batch_number
- expiry_date
- reorder_level

Extra fields sent by frontend but not part of the PharmacyMedicine model used in create endpoint logic:
- mrp
- invoiceNumber
- purchasePrice
- purchaseDate
- branchName

Confirmation about stock field:
- Yes, stock is stored in a field named exactly: stock
- Backend reads and writes med.get("stock", 0)

2. MEDICINE CREATION ENDPOINT

Endpoint path:
- POST /pharmacy/medicines

Where defined:
- backend/main.py
- async def create_medicine(medicine_data: dict = Body(...))

Request body structure actually used by backend:
{
  "name": string,
  "category": string,
  "price": number,
  "stock": number,
  "description": string,
  "manufacturer": string,
  "batch_number": string,
  "expiry_date": string,
  "reorder_level": number

Notes:
- Backend ignores extra fields unless explicitly copied into med dict.
- The create endpoint currently copies only the fields listed above.

Example payload:

  "price": 150,
  "stock": 50,
  "description": "Artificial tears for dry eyes",
  "manufacturer": "Allergan",

- But backend create_medicine persists only:
  name, category, price, stock, description, manufacturer, batch_number, expiry_date, reorder_level, created_at, updated_at

3. MEDICINE STOCK UPDATE LOGIC

Places where stock is modified:

A. Manual stock update endpoint
- PUT /pharmacy/medicines/{medicine_id}/stock
- Expects payload:
  {
    "quantity_sold": int
  }
- Logic:
  1. Load current medicine
  2. Read current_stock = med.get("stock", 0)
  3. Validate current_stock >= quantity_sold
  4. Compute new_stock = current_stock - quantity_sold
  5. Update using $set:
     {
       "$set": {
         "stock": new_stock,
         "updated_at": timestamp
       }
     }

B. Generic medicine update endpoint
- PUT /pharmacy/medicines/{medicine_id}
- Accepts arbitrary medicine_data
- Used by frontend MedicineManagementView for:
  - adding stock
  - removing stock
  - editing medicine details
- Stock change here is done by sending a new absolute stock value, then backend applies:
  {
    "$set": update_data
  }

C. Pharmacy billing stock reduction
- POST /pharmacy/billing
- Before billing is committed, backend validates stock for every item.
- After the bill is inserted, stock is reduced per item using $inc:
  {
    "$inc": {
      "stock": -quantity_sold
    },
    "$set": {
      "updated_at": timestamp
    }
  }

How stock is reduced during pharmacy billing:
- create_pharmacy_bill reads billing_data.items
- Each item must contain medicineId and quantity
- Backend checks available stock first
- If all items have enough stock, it inserts the bill into pharmacy_billing_collection
- Then for each billed item it updates pharmacy_collection with:
  stock = stock - quantity_sold
- This reduction uses $inc, not manual recomputation

4. INVOICE UPLOAD FRONTEND PAYLOAD

Important finding:
- In src/components/InvoiceUploadView.tsx, clicking "Add Stock" does NOT send any request to the backend.
- There is no fetch/axios call in the Add Stock button handler.
- Instead, it constructs a GRN object and calls:
  onSubmit?.(grnEntry)
  onNavigate?.('grn-history')

Conclusion for question 4:

5. MEDICINE IDENTIFIER

Backend database matching for medicine records:
- Primary exact lookup/update field: _id (Mongo ObjectId)
- Fallback exact lookup/update field: id

Observed patterns:
- pharmacy_collection.find_one({"_id": ObjectId(medicine_id)})
- fallback: pharmacy_collection.find_one({"id": medicine_id})
- same pattern used for update_one calls

Frontend matching behavior:
- Invoice import matching in MedicineManagementView uses exact normalized medicine name:
  m.name.toLowerCase().trim() === item.name.toLowerCase().trim()
- This is only for UI-side pre-match during invoice import review
- It is not the primary backend database identifier

Exact field used overall:
- Backend CRUD/billing operations: _id, with fallback to id
- Invoice import pre-match only: name

SUMMARY

- Medicine inventory is stored in collection: pharmacy_medicines
- Stock is stored in field: stock
- New medicines are created through POST /pharmacy/medicines
- Stock is reduced during billing in POST /pharmacy/billing using $inc: { stock: -quantity_sold }
- InvoiceUploadView Add Stock does not directly call backend; it emits a GRN object via onSubmit
- Backend medicine identity is primarily _id, with fallback id
- Pharmacy-related collections are pharmacy_medicines, pharmacy_billing, and patients (embedded pharmacy history)
