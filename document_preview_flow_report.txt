============================================================
5. WHY PATIENT DOCUMENT PREVIEW MAY NOT BE WORKING
============================================================

Frontend behavior is correct for preview intent:
- Preview button uses handlePreview(doc)
- handlePreview(doc) calls getDownloadUrl(doc, true)
- getDownloadUrl(doc, true) resolves to:
  /patients/{registration_id}/documents/{file_id}/download?inline=1
- The frontend then calls window.open(url, '_blank')

So the frontend is not using ?inline=false and is not deliberately forcing a download on preview clicks.

The likely failure is on the backend response headers:
- In inline mode, the backend calls:
  FileResponse(str(path), media_type=media_type, filename=match.get('name'))
- The route does not explicitly set Content-Disposition: inline
- Because filename is supplied, FileResponse commonly defaults to attachment disposition unless content_disposition_type='inline' is explicitly provided

Result:
- Even though the frontend requests ?inline=1 and the backend sets a previewable media_type, the response may still arrive as an attachment, causing the browser to download instead of preview

Additional mismatch that can also affect preview:
- Frontend treats avi and mov as video documents
- Backend inline media-type logic only handles mp4, webm, ogg, pdf, png, jpg, jpeg, gif
- avi and mov are not assigned a previewable media_type in the backend route
- Such files will fall back to application/octet-stream behavior

6. BACKEND PREVIEW SUPPORT
============================================================

A. Patient document upload route

File: backend/main.py
Route:
- POST /patients/{registration_id}/documents
Behavior:
- Verifies patient exists
- Saves files under backend/uploads/{registration_id}/
- Generates UUID fileId and storedName
- Stores metadata in patient_documents collection
- Returns serialized saved metadata
Stored metadata fields include:
- fileId
- registrationId
- name
- storedName
- size
- type
- uploadedDate
- uploadedBy

B. Patient document list route

File: backend/main.py
Route:
- GET /patients/{registration_id}/documents
Behavior:
- Reads from patient_documents collection sorted by uploadedDate descending
- Falls back to legacy patient.documents if collection has no rows
- Returns sanitized serialized metadata

C. Patient document download/preview route

File: backend/main.py
Route:
- GET /patients/{registration_id}/documents/{file_id}/download
Signature:
- download_patient_document(registration_id, file_id, inline: bool | None = None)
Behavior:
- Looks up document in patient_documents collection
- Falls back to legacy patient.documents structure
- Resolves path from storedName or legacy path
- Validates that file exists on disk
- Determines media_type from extension

Inline support:
- Yes, the route accepts ?inline=1
- If inline is truthy, it returns FileResponse with the detected media_type

Exact backend inline logic:
- if ext in ('png', 'jpg', 'jpeg', 'gif'): media_type = image/*
- elif ext in ('mp4', 'webm', 'ogg'): media_type = video/*
- elif ext == 'pdf': media_type = application/pdf
- if inline: return FileResponse(str(path), media_type=media_type, filename=match.get('name'))
- else: return FileResponse(str(path), media_type='application/octet-stream', filename=match.get('name'))

Important limitation:
- The code does not explicitly set Content-Disposition: inline
- Therefore preview support is partial/incomplete even when ?inline=1 is present

D. Slit lamp backend support

File: backend/main.py
Routes:
- POST /slit-lamp/upload
- GET /slit-lamp/patient/{patient_id}
Behavior:
- Stores image as base64 string in MongoDB
- Lists previously stored images for a patient
Notes:
- This is independent from patient document download/preview
- No dedicated download route found because images are stored inline in MongoDB documents
